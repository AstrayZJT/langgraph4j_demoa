# 核心库：概念指南

> 说明：本文是对 LangGraph4j 官方 `Core Library` 页面做的中文整理版，用来作为本项目的教材文档。

## 1. LangGraph4j 在做什么

LangGraph4j 把 agent 工作流建模成“图”。

图里最重要的三件事是：

1. `State`：共享状态，表示当前执行到哪一步、积累了哪些数据。
2. `Node`：执行逻辑的单元。节点读取当前状态，做计算或副作用，再返回状态更新。
3. `Edge`：决定下一步去哪里。边可以是固定跳转，也可以是条件分支。

一句话理解：

- `node` 负责干活
- `edge` 负责指路

这样组合之后，就能很自然地表达循环、分支、暂停、恢复、并行等工作流。

## 2. StateGraph

`StateGraph` 是最主要的图构建器。它会绑定你自己定义的 `State` 类型。

创建图的顺序通常是：

1. 先定义 `State`
2. 再添加 `Node`
3. 再添加 `Edge`
4. 最后 `compile()`

## 3. 为什么要 compile

图不是“写完就直接跑”，而是要先编译。

编译的作用主要有三类：

1. 检查图结构是否合理，比如有没有孤立节点。
2. 绑定运行时配置，比如 checkpoint、断点。
3. 把图变成可以执行的形式。

没有 `compile()`，图不能执行。

### CompileConfig

`CompileConfig` 用来控制图编译后的运行行为。

常见配置包括：

- `checkpointSaver`：持久化状态
- `recursionLimit`：限制循环深度，防止无限循环
- `interruptBefore` / `interruptsAfter`：在指定节点前后中断
- `graphId`：图标识，便于日志和追踪

## 4. 执行图

图编译完成后，有两种常见执行方式。

### 4.1 同步执行

同步执行会一直跑到结束，然后直接返回最终状态。

适合：

- 只关心结果
- 不需要看中间过程

### 4.2 流式执行

流式执行会把每一步的状态变化都吐出来。

适合：

- 观察执行过程
- 前端实时展示
- 调试工作流
- 处理中断和恢复

你当前这个学习项目用的就是这种方式，所以控制台会看到 `NodeOutput` 一步一步打印出来。

## 5. RunnableConfig

`RunnableConfig` 是运行时配置对象。

它可以把一些“每次执行都可能变化”的信息传进图里，例如：

- `threadId`
- `graphId`
- `metadata`
- `streamMode`

### 为什么需要它

因为图结构本身应该保持稳定，但每次运行时的上下文可能不同。

比如：

- 用户是谁
- 当前会话是哪条线程
- 这次用哪个模型
- 这次是否开启某个功能开关

这些都适合放进 `RunnableConfig`。

## 6. GraphResult

流式执行返回的是一串结果，不同步骤可能返回不同类型。

`GraphResult` 的作用就是帮你安全判断和取出结果类型，避免手写一堆 `instanceof`。

常见结果包括：

- 状态数据
- 节点输出
- 中断信息
- checkpoint 标记

## 7. State

`State` 是图最核心的数据结构。

它由两部分组成：

1. schema：有哪些字段
2. reducer：字段收到更新后如何合并

如果某个字段没定义 reducer，默认就是覆盖式更新。

### 7.1 Schema

schema 通常通过 `Channel` 的映射来定义。

每个 key 对应一个状态字段。

### 7.2 Reducer

reducer 决定“节点返回的新值”怎么和旧值合并。

这件事非常重要，因为它决定了你的状态是：

- 覆盖
- 追加
- 累积
- 自定义合并

### 7.3 AppenderChannel

`AppenderChannel` 是内置的追加型通道。

它适合：

- 消息列表
- 历史记录
- 对话内容

它的特点是：节点往里写一条，状态里会保留原有内容并继续追加。

这也是你当前示例里 `messages` 能不断增长的原因。

## 8. Nodes

节点通常是一个函数接口。

它接收当前 `State`，执行逻辑，然后返回一个“状态更新请求”。

节点里既可以写纯 Java 逻辑，也可以调用 LLM、工具、数据库或外部服务。

### START 节点

`START` 是虚拟起点。

它不执行业务逻辑，只是告诉图：从哪里开始。

### END 节点

`END` 是虚拟终点。

当边指向 `END`，图就结束。

## 9. Edges

边定义“下一步去哪里”。

### 9.1 普通边

固定从 A 走到 B。

适合简单串联流程。

### 9.2 条件边

先执行一个路由函数，再根据返回值决定下一步去哪个节点。

适合：

- if / else
- 分流
- 动态路由

### 9.3 Entry Point

决定用户输入进来后先进入哪个节点。

### 9.4 Conditional Entry Point

先判断，再决定从哪个入口开始。

## 10. Checkpointer

LangGraph4j 内置了持久化层，核心就是 `Checkpointer`。

它会在每一步保存图状态，所以可以支持：

- 人工确认后继续执行
- 中断后恢复
- 线程记忆

这也是 LangGraph4j 很适合做 agent 工作流的地方。

## 11. Threads

`Thread` 是 checkpoint 的归属单位。

你可以把它理解成一次会话的 ID。

只要用了 checkpointer，执行图时就需要带上 `threadId`。

它的意义是：

- 区分不同用户会话
- 保存不同运行轨迹
- 支持断点恢复

## 12. Checkpointer state

每个 checkpoint 里，至少会记录两类东西：

1. 当前 `state`
2. 下一步该执行哪个节点

因此恢复时，图不是从头跑，而是能接着继续。

### 12.1 Get state

可以按 `threadId` 取当前状态。

### 12.2 Get state history

可以按 `threadId` 取完整历史。

### 12.3 Update state

可以直接更新状态，但它仍然会走 reducer 逻辑，不是粗暴覆盖。

这点很关键，因为它保证了状态更新规则的一致性。

## 13. 这一页最重要的结论

如果你只记住一句话，那就是：

> LangGraph4j = 用 `State` 保存上下文，用 `Node` 执行业务，用 `Edge` 控制流转。

而 `compile`、`RunnableConfig`、`GraphResult`、`Checkpointer`、`Threads` 则是在这个基础上，把它变成一个能调试、能恢复、能扩展的 agent 工作流引擎。

# 循环

这一节要学的是：**图不只会分支，还可以自己反复跑，直到满足条件为止。**

这一步很接近真实 agent 的行为，因为很多任务都不是“一次就完”，而是：

- 没想好就再想一次
- 没查到就再查一次
- 没通过就再试一次

## 2. 这一步想解决什么

条件分支解决的是“下一步去哪”。

循环解决的是“要不要再走一遍同一段流程”。

比如：

- 一个任务要重试 3 次
- 一个检查没通过就重新执行
- 一个 agent 需要不断修正自己的答案

如果没有循环，这些流程就只能写成一堆手工 `while` 或 `if`。

---

## 3. 这个示例的图长什么样

这次我们做的是一个“倒计数循环”：

```text
START -> counter -> counter -> counter -> END
```

更准确地说：

- `counter` 每执行一次，`remaining` 就减 1
- 如果 `remaining > 0`，就继续回到 `counter`
- 如果 `remaining == 0`，就结束

---

## 4. State 里多了什么

[`LoopState`](../../src/main/java/com/example/langgraph4jdemo/loop/LoopState.java) 里有两个字段：

```java
public static final String REMAINING_KEY = "remaining";
public static final String MESSAGES_KEY = "messages";
```

### `remaining`

这个字段表示还剩几轮。

```java
REMAINING_KEY, Channels.base(() -> 0)
```

它是普通值，每次更新都会覆盖。

### `messages`

还是追加型通道，用来记录每次循环的痕迹。

---

## 5. 图是怎么建的

核心代码在 [`LoopService`](../../src/main/java/com/example/langgraph4jdemo/loop/LoopService.java)：

```java
var stateGraph = new StateGraph<>(LoopState.SCHEMA, LoopState::new)
        .addNode("counter", node_async(countdownNode))
        .addEdge(START, "counter")
        .addConditionalEdges(
                "counter",
                edge_async(state -> state.remaining() > 0 ? "continue" : "end"),
                Map.of(
                        "continue", "counter",
                        "end", END
                )
        );
```

### 这里的核心点

`counter` 节点执行完以后，不是直接结束，而是交给条件边判断：

- 还没到 0，就返回 `"continue"`，再走回 `counter`
- 到 0 了，就返回 `"end"`，走向 `END`

所以循环不是库自己猜出来的，而是你在条件函数里写出来的。

---

## 6. `CountdownNode` 做了什么

[`CountdownNode`](../../src/main/java/com/example/langgraph4jdemo/loop/CountdownNode.java) 每次执行都会：

1. 打印当前剩余次数
2. 把 `remaining` 减 1
3. 追加一条 `tick` 消息

```java
return Map.of(
        LoopState.REMAINING_KEY, state.remaining() - 1,
        LoopState.MESSAGES_KEY, "tick: " + state.remaining()
);
```

这就让你能在控制台里看到每一轮循环都发生了什么。

---

## 7. 为什么要加 `recursionLimit`

循环最怕的就是写错条件，导致死循环。

所以这里在编译时加了一个安全上限：

```java
CompileConfig.builder()
        .graphId("loop-demo")
        .recursionLimit(20)
        .build()
```

它的意义是：

- 正常情况下让图继续跑
- 万一条件写错，也不会无限跑下去

这就是图式工作流里很重要的保护措施。

---

## 8. 程序是怎么跑这个示例的

`LangGraph4jDemoApplication` 启动后，这一步会跑两次：

- `3` -> 循环 3 次
- `1` -> 循环 1 次

这样你能很直观看到“循环多次”和“循环一次”的区别。

### `compiledGraph.stream(...)` 到底是怎么把图跑起来的

这句代码：

```java
for (var item : compiledGraph.stream(Map.of(LoopState.REMAINING_KEY, start))) {
    System.out.println(item);
}
```

不是 `for` 循环自己在“执行图”，而是：

1. 你先把初始输入交给 `compiledGraph.stream(...)`
2. `stream(...)` 返回一个会不断吐出执行结果的对象
3. `for-each` 只是把这些结果一个个取出来打印

### 数据是怎么传进去的

你传入的是：

```java
Map.of(LoopState.REMAINING_KEY, start)
```

比如 `start = 3` 的时候，初始输入就是：

```text
remaining = 3
```

LangGraph4j 会先根据 `LoopState::new` 把这份 Map 包装成状态对象，然后从 `START` 进入图。

### 节点、边、状态是怎么配合的

以 `remaining = 3` 为例：

1. 图进入 `counter`
2. 框架调用你写的 `CountdownNode.apply(state)`
3. 节点返回一个更新 Map：
   ```java
   { remaining = 2, messages = "tick: 3" }
   ```
4. LangGraph4j 按 `Channel` 的规则合并这份更新
5. 条件边判断 `remaining > 0`，如果成立就跳回 `counter`
6. 直到 `remaining == 0`，才走到 `END`

所以：

- **节点函数**负责产出更新
- **Channel** 负责怎么合并更新
- **条件边** 负责决定要不要继续循环

---

## 9. `Channel` 和 `Channels.base(() -> 0)` 是什么

`Channel` 可以理解成“某个 state 字段的更新规则”。

LangGraph4j 不只是要知道字段名，还要知道：

- 默认值是什么
- 新值来了怎么合并
- 是覆盖还是追加

### `Channels.base(() -> 0)`

```java
REMAINING_KEY, Channels.base(() -> 0)
```

这表示 `remaining` 是普通单值字段：

- 默认值是 `0`
- 新更新来了就覆盖旧值

比如：

- 旧值 `3`
- 新值 `2`
- 合并后就是 `2`

### `Channels.appender(ArrayList::new)`

```java
MESSAGES_KEY, Channels.appender(ArrayList::new)
```

这表示 `messages` 是追加型字段：

- 默认是一个空列表
- 新消息来了，不覆盖旧消息，而是追加进去

比如：

- 旧值 `["tick: 3"]`
- 新消息 `"tick: 2"`
- 合并后就是 `["tick: 3", "tick: 2"]`

---

## 10. 为什么这一层很重要

因为你现在看到的不是“一个函数返回一个结果”，而是：

- 节点产出局部更新
- Channel 决定怎么合并
- 图引擎继续下一步

这就是图式工作流真正厉害的地方。

---

## 11. 怎么验证这一步成功

### 3 的时候

你应该看到 `counter` 连续执行 3 次左右，然后结束。

### 1 的时候

你应该看到 `counter` 只执行 1 次，然后结束。

同时 state 里的 `remaining` 会逐步减到 0。

---

## 12. 这一节学完后你应该明白什么

你应该真正理解这些事：

1. 循环不是库自动魔法
2. 是你用条件边把节点重新指回自己
3. `remaining` 这种状态值，决定要不要继续循环
4. `recursionLimit` 是安全保险
